# KRAS DMS × ESM-C SAE feature analysis

This analysis tests whether the features learned by ESM-C SAEs capture the functional effect of mutations, using KRAS as the
test case. For every single amino-acid variant in a KRAS deep-mutational-
scanning (DMS) dataset, it runs the mutant through the ESM-C 6B SAE, measures
how much each SAE feature changes, and compares that against the measured DMS
effect.

This README documents the repository. The scientific report is in [`reports/RESULTS.md`](reports/RESULTS.md).

**Summary of findings.** Across all seven KRAS DMS assays, the SAE disruption
caused by a mutation is significantly larger for damaging substitutions than
for neutral ones; and at mutational hotspots the most-disrupted SAE features
correspond to the known local biology (for example, the phosphate-binding
loop).

## Data

All data lives in [`data/`](data/) and falls into two groups.

**Downloaded inputs** — obtained from external sources, not produced by this
repository:

| File | Source | Used by |
|---|---|---|
| `41586_2023_6954_MOESM5_ESM.xlsx`, `…MOESM6_ESM.xlsx` | Weng et al., *Nature* 2024 — KRAS DMS supplementary tables | steps 01, 03 |
| `6vjj.pdb` | RCSB PDB — the KRAS–RAF1 complex structure | step 04 |
| `uniref90_feature_table.parquet` | the ESM-C SAE feature dictionary: 16,384 rows, one label per SAE feature | steps 08, 09, 11 |

**Generated by the pipeline** — produced by the scripts, not supplied by the
user:

| File | Produced by | Used by |
|---|---|---|
| `kras_sae_features.pt` | step 01 (ESM-C SAE via the Forge API) | step 02 |
| `dms_matrices/kras_sae_tensor*.pt`, `…_meta.json` | step 02 | steps 05–11 |
| `dms_matrices/kras_ddG_*.csv`, `kras_fitness_*.csv`, `kras_wt.fasta` | step 03 | steps 05–11 |
| `kras_anno.csv` | step 04 | step 10 |

The three SAE tensor files — `kras_sae_features.pt` and the two
`dms_matrices/kras_sae_tensor*.pt` — each exceed GitHub's 100 MB per-file limit
and so are not committed. `kras_sae_features.pt` is the output of the API step
and the input the rest of the pipeline depends on: obtain it from the authors,
or regenerate it by running step 01. The `dms_matrices` tensors are then
rebuilt from it by step 02 in under a minute. Everything else in `data/`, and
all code, is committed.

`results/` is committed, with one exception: `results/validation_3/` holds the
supplementary within-position control (script 11), which is not part of this
release and is not committed.

The DMS dataset comprises seven assays — one folding readout and six binding
readouts (`RAF1`, `PIK3CG`, `RALGDS`, `SOS1`, `DARPin K27`, `DARPin K55`). The
report describes what each measures.

## Pipeline

Eleven numbered steps under [`scripts/`](scripts/). Step 01 is the only API
step; steps 02–11 are CPU-only and rebuild every result and figure.

| Step | Script | Does |
|---|---|---|
| 01 | `01_compute_sae_features.py` | ESM-C 6B SAE features for every DMS variant (Forge API) |
| 02 | `02_export_sae_tensors.py` | reshape into the per-mutant SAE tensor (20 × 187 × 16384) |
| 03 | `03_export_dms_matrices.py` | per-assay effect matrices in heatmap layout |
| 04 | `04_compute_structural_annotations.py` | per-residue interface / pocket / core / secondary structure from PDB 6VJJ |
| 05 | `05_validation1_global.py` | Validation 1 — global SAE disruption vs effect, per-assay statistics |
| 06 | `06_validation1_aggregations.py` | Validation 1 — per-aggregation figure grids |
| 07 | `07_validation1_sweep.py` | Validation 1 — robustness sweep |
| 08 | `08_validation2_hotspot_enrichment.py` | Validation 2 — per-cluster permutation test |
| 09 | `09_validation2_descriptive.py` | Validation 2 — descriptive per-cluster feature ranking |
| 10 | `10_validation2_plot_callouts.py` | Validation 2 — callout heatmap figures |
| 11 | `11_validation3_within_position.py` | within-position control (supplementary; not in the report) |

`scripts/config.py` resolves every path relative to this folder;
`scripts/dms_common.py` holds the hotspot-clustering helpers shared by steps
08 and 09.

## Running the pipeline

Steps 02–11 are CPU-only. Given `data/kras_sae_features.pt` (see Data above),
they rebuild every result and figure:

```bash
cd scripts
for s in 02 03 04 05 06 07 08 09 10 11; do python ${s}_*.py; done
```

A full re-run takes a few minutes; step 08's 200,000-permutation test
dominates, at about 2.5 minutes. `scripts/config.py` resolves every path
relative to the repository, so no machine-specific configuration is needed.

Step 01 (`01_compute_sae_features.py`) calls the EvolutionaryScale Forge API,
which consumes credits, and is not part of a routine re-run. Running it
requires the EvolutionaryScale `esm` package and `cookbook` snippets
importable, and `ESM_API_KEY` set to a Forge token:

```bash
export ESM_API_KEY=<your Forge token>
python scripts/01_compute_sae_features.py
```

## Skills

[`skills/`](skills/) holds reusable, protein-agnostic workflows distilled from

## Dependencies

Python ≥ 3.8 with `torch`, `numpy`, `pandas`, `pyarrow`, `scipy`,
`statsmodels`, `matplotlib`, `biopython`, and `freesasa`. Step 01 additionally
requires the EvolutionaryScale `esm` package and Forge API access.
